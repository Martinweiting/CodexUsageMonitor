# CodexUsageMonitor 資產化玻璃介面與懸浮互動設計

## 1. 目標

本階段集中處理四個目前使用上的問題：

1. hover 展開時，視窗尺寸變更造成游標事件失真，導致圖示與完整面板反覆切換或面板無法操作。
2. 懸浮球與重新整理控制項必須使用指定的 PNG 資產，而不是目前的向量仿製圖形。
3. 所有顯示模式需要更接近有厚度的液態玻璃，並降低目前固定淡藍色矩形的平面感。
4. 完整模式資訊過度擁擠，需要重新分組、增加留白，且完整面板必須顯示在懸浮球左側、與懸浮球完全分離。

本階段保留既有用量計算、帳號處理、資料讀取、重新整理、計時器、語言內容、拖曳、置頂、鎖定位置、顯示模式與設定儲存流程。修改集中在視窗互動、渲染資產、字型、幾何與視覺層。

## 2. 現況與根因

目前程式使用單一 `WS_EX_LAYERED` Win32 視窗，將 Direct2D 畫面渲染到 32-bit DIB，再透過 `UpdateLayeredWindow` 顯示。懸浮狀態由 `Bubble`、`HoverExpanded`、`PinnedExpanded` 三種狀態表示。

目前 hover 流程在 `WM_MOUSEMOVE` 中立即呼叫 `SetPresentationState`，而 `SetPresentationState` 同時改變視窗位置與大小。原本以 `TrackMouseEvent(TME_LEAVE)` 建立的滑鼠追蹤可能仍對應到尺寸變更前的視窗區域，因此 Windows 可能在重定位後送出過期的 `WM_MOUSELEAVE`。這會讓狀態剛展開就被收合，再次收到滑鼠移入後又重新展開。

目前懸浮球與展開面板共用展開面板右上方區域，造成圖示覆蓋面板內容。渲染中的 `drawGlassIcon` 與重新整理符號也是向量近似版本，尚未使用指定 PNG。

## 3. 互動與狀態設計

### 3.1 單一視窗狀態機

維持單一 layered window 與現有三種展示狀態：

- `Bubble`：只顯示指定的 Codex PNG 懸浮球。
- `HoverExpanded`：滑鼠停留在懸浮球或目前完整面板範圍內時顯示完整內容；移出後立即回到 `Bubble`。
- `PinnedExpanded`：使用者點擊懸浮球後固定顯示；移出不收合，直到再次點擊懸浮球或右上角 `×`。

### 3.2 穩定的 hover 事件

新增集中管理滑鼠追蹤的輔助流程：

1. 展示狀態改變前取消舊的 `TrackMouseEvent` 離開追蹤。
2. 完成 `MoveWindow`／`UpdateLayeredWindow` 後，依新視窗矩形重新建立追蹤。
3. `WM_MOUSELEAVE` 不直接改狀態，先以 `GetCursorPos` 取得螢幕游標，再判斷游標是否真的離開目前視窗矩形；若事件只是視窗重定位造成的過期事件，忽略它。
4. 加入短時間的離開檢查計時器作為事件佇列保護。保護時間只用來消除重定位期間的假離開，不改變使用者移開後立即收合的語意。
5. `PinnedExpanded` 忽略所有 hover 收合事件。
6. 點擊控制項優先於拖曳判斷；點擊懸浮球使用既有狀態轉移 helper，避免 `Bubble` 狀態點擊變成無作用。

### 3.3 完整面板與懸浮球的群組幾何

以目前儲存的完整面板矩形作為主要定位來源，不重新解讀既有的 x、y、width、height 設定。新增固定間距 `kPanelBubbleGap`，建議值為 14 個邏輯像素。

- 完整面板保持在儲存矩形的位置。
- 懸浮球位於完整面板右側，兩者之間保留 `kPanelBubbleGap`。
- 展開視窗的外框包含「完整面板 + 間距 + 懸浮球」，但兩個可視表面不重疊。
- 收合時只顯示同一群組右側的懸浮球。
- 拖曳懸浮球或面板時移動整個群組，保持面板與懸浮球的相對位置。
- 群組接近螢幕邊界時，優先整體平移；必要時依目前螢幕工作區限制面板尺寸，但不以重疊方式解決邊界問題。
- 在任務模式中同樣保留間距，窄版內容仍位於懸浮球左側。

hover 放大只改變懸浮球繪製區域與群組的最小外框，不改變完整面板的儲存尺寸與內容排版。若目前螢幕高度不足，放大尺寸依 DPI 與可用高度限制。

## 4. 指定資產與字型

### 4.1 資產目錄

將使用者提供的檔案整理為：

```text
assets/
  icons/
    codex-liquid.png
    undo.png
  fonts/
    Iansui-Regular.ttf
    Quantico-Regular.ttf
    Quantico-Italic.ttf
    Quantico-Bold.ttf
    Quantico-BoldItalic.ttf
```

附件 1 複製為 `codex-liquid.png`，保留透明背景與原始 RGBA。附件 2 保留原始 alpha 與輪廓；因其圖形 RGB 為黑色，渲染時以 Direct2D opacity mask 著色為白色或淡藍色，以確保在玻璃面板上清楚可見，而不重畫或替換圖形。

### 4.2 載入方式

使用 WIC decoder 將 PNG 轉成 Direct2D 可用的 premultiplied BGRA bitmap。Bitmap 資產在裝置資源建立成功後載入，裝置資源失效時一併釋放並重新建立。

執行檔會從自身目錄下的 `assets` 載入。CMake target 會在建置後複製資產目錄到執行檔輸出資料夾；直接 `build.cmd` 建置則使用工作區根目錄的 `assets`。資產不存在或載入失敗時，程式仍可啟動，使用低調的既有幾何 fallback，避免因視覺資產問題影響用量資料功能。

### 4.3 字型

啟動時以 `AddFontResourceExW(..., FR_PRIVATE)` 註冊資產目錄內的字型，結束時解除註冊。DirectWrite text format 建立時：

- 繁體中文介面使用 `Iansui`。
- 英文介面使用 `Quantico`。
- `Iansui` 沒有提供的粗體由 DirectWrite 使用合成粗體；Quantico 依文字權重使用對應的 Regular／Bold 字型檔。
- 混合字串中的拉丁字母、數字與符號優先使用 Quantico；中文文字使用 Iansui。若 DirectWrite 文字 layout 無法分段，至少依目前語言選擇正確的主要字型，並保留系統 fallback。

字型載入失敗時回退到目前系統字型，不阻斷資料讀取或視窗建立。

## 5. 液態玻璃渲染

### 5.1 原生背景模糊

保留目前 Direct2D layered surface 作為主要繪製管線，並在視窗建立時嘗試啟用 Windows 原生 composition acrylic／blur-behind。原生 API 以動態解析方式呼叫，避免在不支援的 Windows 版本上造成啟動失敗。若 layered window 與系統 backdrop 組合在目前環境不可用，渲染器使用同一套材質分層 fallback，不擷取自身畫面，也不引入高成本的桌面截圖模糊。

原生模糊不可用時的 fallback 仍必須保留後方色彩：降低固定藍色遮罩的 alpha，使用中性淡紫染色、透明度漸層與局部高光，而不是單一不透明底色。

### 5.2 三種材質層級

1. 外層玻璃視窗：較低不透明度、較大的圓角、柔和陰影、上／左亮而下／右暗的邊緣漸層。
2. 次級資訊卡／群組：比外層稍微不透明，使用淡紫或淡藍的局部染色，仍保留背景穿透；卡片之間使用細微半透明分隔線。
3. 內容層：文字、圖示、數值、進度條填色、滑桿控制點與重新整理圖示完全不透明，確保高對比與可讀性。

外層邊框使用非均勻亮度：上方與左側加入細微高光，右側與下方降低亮度；邊框內側加入一條低 alpha 反光線，外側使用柔和陰影。資訊卡不使用完全一致的白色描邊。

### 5.3 元件狀態

- 進度條軌道使用半透明低對比玻璃，填色使用完全不透明的狀態色，預算標記使用完全不透明細線。
- 分隔線使用低 alpha、單像素或 DPI 對應的細線。
- 重新整理圖示使用附件 2 的 alpha mask，不使用實心按鈕外框；hover／按下時只提升圖示亮度與外圈微光。
- 懸浮球使用附件 1 的原始透明輪廓；hover 時提升尺寸、亮度與陰影，不添加方形背景。
- 所有玻璃表面維持圓角與連續曲線，避免卡片像白色實心方塊貼在主面板上。

## 6. 完整模式重排

完整模式提高最小尺寸與留白，維持資料完整但減少單一長串文字堆疊。建議基準為約 440 個邏輯像素寬、650 個邏輯像素高，並依目前螢幕工作區與 DPI 調整。

排版順序：

1. 頂部標頭：Codex 資產圖示、帳號／電子郵件、方案與有效期限。
2. 限額摘要雙卡：5 小時與本週各自顯示剩餘百分比、狀態與重設倒數。
3. 週期速度卡：週期第幾天、實際使用量、預算使用量與高於／低於預算差值。
4. 使用量限制重設卡：保留標題 `使用量限制重設 (Full reset Weekly + 5 hr)`，顯示可用數量與每張卡的到期時間，只供檢視。
5. 用量詳情卡：5 小時與週限額各一列，包含開始、重設、預計用完時間、剩餘進度與預算標記。
6. 底部更新列：無框 undo 圖示、版本資訊、上次／下次重新整理倒數。

同一資料只在必要的位置顯示一次；摘要卡負責快速判讀，詳情卡負責時間與進度細節。簡單模式與任務模式保留既有資訊密度，只套用相同的玻璃材質、指定字型與資產圖示。

## 7. 選單與設定

右鍵選單中將目前顯示的 `設定…`／`Settings...` 改為：

- 繁體中文：`透明度`
- 英文：`Transparency`

內部 command ID 不變。透明度設定頁仍使用現有橫向滑桿與儲存鍵值，不改變其 20%–80% 範圍或設定檔相容性。

## 8. 測試與驗收

### 8.1 自動測試

- 擴充 `WidgetPresentationTests`：驗證面板／間距／懸浮球群組矩形、螢幕邊界平移、hover 放大尺寸與三種狀態轉移。
- 擴充來源契約檢查：確認指定資產路徑、字型路徑、`透明度` 選單文字與重設功能仍為唯讀。
- 保留既有用量摘要、重設標題與透明度滑桿測試。

### 8.2 建置與實際驗收

使用 Visual Studio 18 Build Tools 執行：

```powershell
cmake -S . -B build-tests -G "Visual Studio 18 2026" -A x64 -DCODEX_USAGE_MONITOR_BUILD_TESTS=ON
cmake --build build-tests --config Debug
ctest --test-dir build-tests -C Debug --output-on-failure
cmake --build build-tests --config Release
```

手動驗收至少包含：

1. 收合時只看到附件 1 的去背懸浮球，沒有方形底。
2. hover 時懸浮球放大，完整面板出現在左側且與懸浮球有間距，不閃爍、不互相覆蓋。
3. 面板可點擊、可拖曳，移出後 hover 狀態收合；固定狀態移出不收合。
4. 再次點擊懸浮球與 `×` 都能回到收合狀態。
5. 完整模式可讀取全部資訊，視覺層級能辨識外層玻璃、次級卡片與不透明內容。
6. 重新整理圖示是附件 2 的輪廓且無外框；點擊仍觸發既有重新整理流程。
7. 繁體中文使用 Iansui、英文使用 Quantico；資產不存在時仍能啟動。
8. 透明度滑桿、右鍵 `透明度` 選單與既有設定儲存仍正常。

## 9. 非目標

- 不更換用量 API、帳號驗證、JSON 解析、計時器或資料模型。
- 不新增與本次視覺／互動問題無關的功能。
- 不將應用程式改寫成多視窗或 Web UI。
- 不以重新繪製附件圖形取代指定 PNG 資產。
